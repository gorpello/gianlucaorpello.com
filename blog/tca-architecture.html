SwiftUI + The Composable Architecture: Lessons from Production

June 18, 2025 · 8 min read

A year ago, I started building Defenso.it — a LegalTech platform for automating complex legal workflows — using SwiftUI and The Composable Architecture (TCA). We'd just left the Apple Developer Academy, we had strong opinions about state management, and TCA seemed like the principled choice.

It was. But not in the ways I expected. Here's an honest debrief: the patterns that made everything click, the friction we hit in production, and what I'd tell myself on day one.

Why TCA in the first place

When your domain has the complexity of legal workflows — branching case states, multi-party consent chains, document versioning — you need state management that makes impossible states unrepresentable. SwiftUI's @State and ObservableObject scale fine for simple screens, but they don't give you the structure to reason about side effects at feature boundaries.

TCA forces you to think in terms of State, Action, and Reducer — a strict unidirectional data flow where every change to your app's state is an explicit, testable transformation. This isn't philosophy for its own sake. When a bug surfaces at 2am, having every state transition logged and replayable is worth its weight in gold.

The core promise of TCA: every feature is a pure function (State, Action) → (State, Effect). Your business logic becomes deterministic, fully testable, and composable with other features via Scope and ifLet.

What clicked immediately

Composing features

TCA's feature composition is genuinely elegant. When you model a complex screen as a parent reducer that owns child reducers via Scope, the relationship between features becomes explicit at the type level. Our document review flow owns a DocumentViewerFeature, a SignatureFeature, and an AuditLogFeature — and their interactions are mediated entirely through parent Action cases.

struct ReviewFeature: Reducer {
  struct State: Equatable {
    var document: DocumentViewerFeature.State
    var signature: SignatureFeature.State
    var auditLog: AuditLogFeature.State
  }

  enum Action {
    case document(DocumentViewerFeature.Action)
    case signature(SignatureFeature.Action)
    case auditLog(AuditLogFeature.Action)
    case submitForReview
  }

  var body: some ReducerOf<Self> {
    Scope(state: \.document, action: \.document) {
      DocumentViewerFeature()
    }
    Scope(state: \.signature, action: \.signature) {
      SignatureFeature()
    }
    Reduce { state, action in
      switch action {
      case .signature(.signed):
        return .run { send in
          await send(.auditLog(.record(.signatureAdded)))
        }
      // ...
      }
    }
  }
}

Reading this six months later, the intent is still obvious. That doesn't happen with ad-hoc ViewModels.

Testing side effects

TCA's TestStore lets you exhaustively assert on every state change and every effect. For a legal app where the cost of a bug is measured in client liability, this matters enormously.

@Test func documentSubmission() async throws {
  let store = TestStore(
    initialState: ReviewFeature.State(/* ... */)
  ) {
    ReviewFeature()
  } withDependencies: {
    $0.documentClient.submit = { _ in .success(.submitted) }
  }

  await store.send(.submitForReview) {
    $0.isSubmitting = true
  }
  await store.receive(\.submissionResponse.success) {
    $0.isSubmitting = false
    $0.status = .submitted
  }
}

The test reads like a specification. Fail to handle a case? The test tells you exactly which state mutation you missed.

Where TCA creates friction

Boilerplate at the boundary

Every action needs a case. Every state mutation needs to go through the reducer. For simple UI interactions — toggling a disclosure group, tracking focus — this feels like ceremony. TCA 1.x's @ObservableState helps significantly here, but the mindset shift takes time.

Our rule of thumb: any interaction that doesn't touch shared state or trigger a network call lives in @State inside the view. TCA for business logic, local state for pure UI.

Navigation is genuinely hard

Navigation in TCA was the roughest part of our first three months. The stack-based navigation introduced in TCA 1.x is expressive, but it requires you to model your entire navigation graph in state — which forces decisions about architecture that you might not be ready to make.

The most important thing TCA taught me isn't about reducers. It's that navigation is state, and pretending otherwise will catch up with you.

We ended up spending a week redesigning our navigation model when we realised our initial approach made deep-linking impossible. Build the navigation model before the views. Always.

The learning curve for the team

I was the only one on the team with prior TCA exposure. Getting two junior engineers up to speed took longer than I budgeted. The concepts are sound but the vocabulary is unfamiliar, and the framework's own documentation has grown dramatically — but not always in the direction of "here's how to think about this problem."

What helped most: pair programming on the first three features, an internal Notion doc mapping our domain concepts to TCA primitives, and a rule that any new pattern introduced to the codebase needed a worked example in a playground before merging.

Patterns that made everything click

Dependencies as first-class citizens

TCA's @Dependency system (from the swift-dependencies library) solved our testability problems cleanly. Every external system — the document API, the signature service, the audit logger — is a protocol-shaped dependency injected at the store level.

extension DependencyValues {
  var documentClient: DocumentClient {
    get { self[DocumentClient.self] }
    set { self[DocumentClient.self] = newValue }
  }
}

// In tests:
store = TestStore(/* ... */) {
  DocumentReviewFeature()
} withDependencies: {
  $0.documentClient = .mock
  $0.clock = .immediate
}

No singleton injection, no environment objects threading through five view layers. The dependency graph is explicit and swappable.

Using Equatable state aggressively

Every State conforming to Equatable means SwiftUI only re-renders when state actually changes. For performance-sensitive views — our document list can have hundreds of entries — this is essential. Wrap any non-equatable types in an EquatableNoop wrapper or box them behind an ID.

Would I choose TCA again?

Yes — but with clearer eyes. TCA is the right tool for features with complex state, multiple side effects, and strict testability requirements. It's over-engineered for simple CRUD screens.

The pattern I'd recommend: start with TCA from day one for your core business-logic features, use simpler @Observable models for peripheral screens, and resist the urge to TCA-ify everything just for consistency. Consistent outcomes matter more than consistent architecture everywhere.

The real win isn't the architecture itself — it's the discipline it enforces. When every state change is explicit and every effect is declared, your codebase becomes a readable specification of what your app does. In a domain as consequential as legal workflows, that's not optional.
